üîç Vulnerability Management

Vulnerability Management: A crucial cybersecurity process that plays a vital role in identifying, prioritizing, and remediating vulnerabilities in technical environments. It uses vulnerability scanning to detect new vulnerabilities and implements a remediation workflow addressing highest-priority issues.

Identifying Scan Targets

Organizations must determine which systems to include in vulnerability scans based on several factors:

  • Data classification of information stored, processed, or transmitted
  • Internet or public network exposure
  • Services offered by the system
  • System type (production, test, or development)
Asset Inventory & Criticality: Organizations use scanning tools to build asset inventories and supplement with system type and data handling information. This guides scan frequency, types, and remediation priority decisions.

Determining Scan Frequency

Multiple factors influence scan frequency decisions:

Factor Impact on Frequency
Risk Appetite Risk-averse organizations scan more frequently
Regulatory Requirements PCI DSS, FISMA may dictate minimum frequencies
Technical Constraints Scanner capacity may limit frequency
Business Constraints Avoid disruption during critical periods
Licensing Limitations May restrict bandwidth or simultaneous scans

Configuring Vulnerability Scans

Scan Sensitivity Levels

Administrators should customize scan sensitivity to balance effectiveness with system stability. Templates (vendor-provided or custom) help ensure consistent configurations and efficient reuse.

Supplementing Network Scans

  • Credentialed Scans: Use provided credentials to access target servers for more accurate vulnerability detection
  • Agent-based Scanning: Install software agents on target servers for "inside-out" vulnerability assessment
Exam Note: Credentialed scans typically only retrieve information and don't make changes. Apply principle of least privilege by providing read-only scanner accounts.

Scan Perspectives

Scan Type Purpose Viewpoint
External Internet-based attack simulation Outside attacker perspective
Internal Corporate network assessment Malicious insider perspective
Datacenter Most accurate vulnerability view Direct system assessment
Intrusive vs. Non-intrusive Plug-ins: Some plug-ins may disrupt production systems or damage content. Use test environments first, or maintain non-intrusive scanning on production systems.

Scanner Maintenance

  • Scanner Software: Regular patching protects against scanner vulnerabilities and provides bug fixes
  • Vulnerability Plug-in Feeds: Configure daily automatic updates for new vulnerability detection capabilities

üîí Security Content Automation Protocol (SCAP)

SCAP: NIST-led effort creating standardized approaches for communicating security-related information, enabling automation between security components.

SCAP Components

Component Purpose
CCE Common Configuration Enumeration - system configuration issues
CPE Common Platform Enumeration - product names and versions
CVE Common Vulnerabilities and Exposures - security-related software flaws
CVSS Common Vulnerability Scoring System - vulnerability severity measurement
XCCDF Extensible Configuration Checklist Description Format
OVAL Open Vulnerability and Assessment Language - low-level testing procedures
Exam Note: CVE was previously "Common Vulnerability Enumeration" but is now "Common Vulnerabilities and Exposures." Both terms refer to the standard naming system for flaws.

üìä Understanding CVSS

CVSS: Industry standard for assessing security vulnerability severity, providing scores from 0-10 based on eight different measures (four exploitability, three impact, one scope).

Attack Vector (AV) Physical(P), Local(L), Adjacent(A), Network(N) How attacker exploits vulnerability
Attack Complexity (AC) High(H), Low(L) Difficulty of exploitation
Privileges Required (PR) High(H), Low(L), None(N) Account access needed
User Interaction (UI) None(N), Required(R) Need for human involvement
Scope (S) Unchanged(U), Changed(C) Affects beyond vulnerability scope
Confidentiality (C) None(N), Low(L), High(H) Information disclosure impact
Integrity (I) None(N), Low(L), High(H) Information alteration impact

üéØ Penetration Testing

Penetration Testing: Authorized, legal attempts to defeat an organization's security controls and perform unauthorized activities, bridging the gap between automated tools and skilled attacker capabilities.

The Hacker Mindset

Key Principle: Cybersecurity professionals must win every time; attackers need to win only once. Penetration testers adopt the attacker's perspective - finding a single exploitable vulnerability rather than evaluating all controls.

Penetration Test Types

Test Category Focus Objective
Physical Physical security controls Assess facility access prevention
Offensive Proactive vulnerability exploitation Simulate real-world cyberattacks
Defensive Defense capability evaluation Assess detection and mitigation effectiveness
Integrated Combined offensive/defensive Comprehensive security posture assessment

Knowledge Classifications

Test Type Information Provided Advantage Limitation
Known Environment Full system details Complete, efficient testing May not reflect external attacker view
Unknown Environment No information Realistic attacker simulation Time-consuming, tester skill dependent
Partially Known Environment Limited information Balances efficiency and realism Moderate limitations of both approaches

Penetration Testing Process

  1. Reconnaissance: Information gathering (passive and active)
  2. Initial Access: Exploit vulnerabilities to gain entry
  3. Privilege Escalation: Increase access levels
  4. Lateral Movement/Pivoting: Expand access to other systems
  5. Persistence: Maintain access mechanisms
  6. Cleanup: Remove tools and restore systems
Reconnaissance Types:
  • Passive: No direct target engagement (WHOIS, web searches, public websites)
  • Active: Direct target engagement (port scanning, footprinting, vulnerability scanning)

üîç Threat Hunting

Threat Hunting: Proactive search for evidence of successful attacks based on the "presumption of compromise" philosophy, using attacker mindset to identify attack artifacts.
Difference from Penetration Testing:
  • Penetration Testing: Tests security controls by simulating attacks
  • Threat Hunting: Searches for evidence of actual successful attacks

Type Purpose Performer Audience

Audit Types

  • Internal Audits: Performed by organization's internal audit staff
  • External Audits: Performed by independent third-party firms
  • Independent Third-Party Audits: Initiated by regulators or other outside entities

Big Four Audit Firms

  • Ernst & Young
  • Deloitte
  • PricewaterhouseCoopers (PwC)
  • KPMG

SSAE 18 and SOC Audits

SSAE 18: Statement on Standards for Attestation Engagements document 18, providing common standards for Service Organization Controls (SOC) audits to reduce multiple third-party audit burden.

üìä Vulnerability Life Cycle

Five Stages of Vulnerability Management

  1. Identification: Discover vulnerabilities through scans, penetration tests, responsible disclosure, or audits
  2. Analysis: Confirm, prioritize, and categorize using CVSS/CVE and organizational factors
  3. Response and Remediation: Apply patches, implement compensating controls, or accept risk
  4. Validation: Verify vulnerability resolution through rescanning or auditing
  5. Reporting: Communicate findings, actions, and recommendations to stakeholders
Vulnerability Response Options:
  • Patching
  • Insurance
  • Segmentation
  • Compensating controls
  • Exceptions and exemptions
